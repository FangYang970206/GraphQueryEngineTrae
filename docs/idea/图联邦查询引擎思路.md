# 图联邦查询引擎技术方案说明书 

## 一、 系统架构与定位 
本系统定位于客户端（应用层）与底层数据源（TuGraph / 外部服务）之间的中间件/代理层。它将 Cypher 语言视为统一的数据访问协议，通过抽象语法树（AST）解析、重写、分发，最终在内存中将异构数据融合成统一的图结构返回。 

核心处理流 (Pipeline)：
- 解析层 (Parser)：接收 Cypher 文本，使用 ANTLR4 生成 AST。
- 计划生成层 (Planner & Rewriter)：基于元数据（区分物理/虚拟），结合预设约束，将 AST裂变为内部执行计划（包含物理 Cypher 和外部 API 任务）。 
- 联邦执行层 (Federated Executor)：Java 异步协程引擎，负责请求TuGraph和外部服务。 
- 结果聚合层 (Result Stitcher)：内存数据对齐、路径（Path）拼接、全局排序与分页。

## 二、 核心约束与前提假设 
为保证系统可落地且不陷入分布式图计算的"长尾复杂度"，系统严格遵循以下约束： 
- 边界约束：虚拟边/虚拟节点仅允许出现在图遍历路径的第一跳或最后一跳。不允许出现 [物理]->[虚拟]->[物理] 的夹心结构。 
- 读写分离约束：联邦查询引擎仅支持只读查询 (MATCH, RETURN等)。任何包含写操作 (CREATE, MERGE, DELETE) 的语句若包含虚拟元素，直接抛出语法不支持异常。
- 外部数据源会与内部数据源通过虚拟边进行关联，内外点通过属性进行映射；
- 整个系统使用Java语言进行编写，不需要实现完整系统，你最终会以SDK的形式存在，输入为cypher，输出结果为查询得到的json响应；
- 关于外部数据源的接口请求，只需要设立好外部数据源的统一查询接口（`DataSourceAdapter`）即可，测试时通过 `MockExternalAdapter` 模拟所有数据源（包括 TuGraph 和外部服务）返回，无需具体实现真实数据源适配器；Mock 代码位于 `src/test/java/.../adapter/MockExternalAdapter.java`； 
- 项目使用spring framework，不需要springboot那么厚重；
- 示例所有的类型都是随机定义的，代码不应写死，UT按照示例mock schema；

## 三、测试场景 

所有示例的返回需要遵从TuGraph的返回格式，不要随意捏造；

### 示例1 
cypher：MATCH (ne:NetworkElement {name: 'NE001'})-[r:NEHasLtps|NEHasAlarms|NEHasKPI]->(target) return ne,target; 
其中NEHasKPI和NEHasAlarms是虚拟边，我需要在拿出来单独放在服务去外部请求，剥离后的cypher： 
1.MATCH (ne:NetworkElement {name: 'NE001'})-[r:NEHasLtps]->(target) return ne,target; 

2.MATCH (ne:NetworkElement {name: 'NE001'}) return ne; 
得到ne需要进一步通过KPI算子和Alarm算子去请求外部数据 最后将上述结果进行汇总，最终结果应包含Ltp、Alarm和KPI；

返回格式示例：

```json
[
  {
    "ne": { "name": "NE001", "type": "NetworkElement" },
    "target": { "id": "LTP1", "type": "Ltp" }
  },
  {
    "ne": { "name": "NE001", "type": "NetworkElement" },
    "target": { "id": "Alarm1", "type": "Alarm" }
  },
  {
    "ne": { "name": "NE001", "type": "NetworkElement" },
    "target": { "id": "KPI1", "type": "KPI" }
  }
]
```

### 示例2 
match p=(ne:NetworkElement {name: 'NE001'})-[:NEHasLtps]->(ltp)-[:LTPHasKPI2]->(target) return p union MATCH p = (ne:NetworkElement {name: 'NE001'})-[:NEHasLtps|NEHasKPI]->(target) return p; 
这是一条返回path的语句，其中LTPHasKPI2和NEHasKPI为外部关系；而剥离后的语句为：
1.首先查询match p=(ne:NetworkElement {name: 'NE001'})-[:NEHasLtps]->(ltp) return ltp; 
2.然后根据ltp的结果查询LTPHasKPI2的关系，得到KPI2的结果；以此得到union上半部分的path集合；
3.查询MATCH p = (ne:NetworkElement {name: 'NE001'})-[:NEHasLtps]->(target) return p; 
4.查询MATCH p = (ne:NetworkElement {name: 'NE001'}) return ne; 根据ne查询NEHasKPI，最终得到p； 汇总上述所有结果得到全部path集合；

返回格式示例：

```json
      {"type": "edge", "label": "NEHasLtps", "props": {}},
      {"type": "node", "label": "Ltp", "props": {"id": "LTP1"}},
      {"type": "edge", "label": "LTPHasKPI2", "props": {}},
      {"type": "node", "label": "KPI", "props": {"id": "KPI_A"}}
    ],
    [
      {"type": "node", "label": "NetworkElement", "props": {"name": "NE001"}},
      {"type": "edge", "label": "NEHasLtps", "props": {}},
      {"type": "node", "label": "Ltp", "props": {"id": "LTP1"}}
    ],
    [
      {"type": "node", "label": "NetworkElement", "props": {"name": "NE001"}},

      {"type": "node", "label": "KPI", "props": {"id": "KPI_B"}}

```

### 示例3 
USING SNAPSHOT('latest', 1) ON ['Card'] MATCH path = (p:Person)-[r1:HAS_CHILD]->(p1)-[r2:HAS_CHILD]->(p2)-[r3:BORN_IN]->(c:Card) RETURN path PROJECT BY {'Card': ['name']} 

其中BORN_IN是虚拟边，Card是外部数据源，USING SNAPSHOT('latest', 1) ON ['Card']表示Card的时间条件，PROJECT BY {'Card': ['name']}是针对CARD返回字段的投影； 

### 示例4 
match (card:Card {name: 'card001'}) where card.resId='2131221'; 
其中Card是外部数据源，该请求则只是请求外部数据，无需剥离cypher； 

### 示例5 
match (card:Card {name: 'card001'})-[r1:CardLocateInNe]->(ne) return ne; where card.resId='2131221'; 
其中Card是外部数据源，该请求先请求外部数据Card，然后根据Card的关联字段，生成cypher去请求ne； 

## 四、 核心架构保障机制 (非功能性设计)
为了使系统具备生产可用性，必须在设计层面保障以下三个维度：

### 4.1 性能设计 (Performance) 

1. **防 N+1 查询风暴（Batching）**：   * 绝对不允许在 `for` 循环中逐个请求外部 KPI/Alarm 接口。必须提取所有涉及的起点 ID 集合（如 1000 个 `ltp.id`），封装成 `IN [id1, id2...]` 的批量请求一次性调用外部算子。 
2. **执行计划缓存 (Plan Cache)**：   * 编译 ANTLR AST 并进行 Visitor 遍历非常消耗 CPU。系统引入 Caffeine 。对输入的 Cypher 字符串生成 Hash，如果命中缓存，直接跳过解析层，复用预先生成的 `ExecutionPlan` 对象。 
3. **并行 IO 调度**：   * 示例中的 `UNION` 左右两段查询在逻辑上完全独立，Java 执行引擎应使用线程池（`ExecutorService`）并发执行多个 `QueryPart`。 

### 4.2 可靠性设计 (Reliability) 

1. 外部依赖策略： 外部系统（如 KPI 监控平台）可能响应缓慢甚至宕机。调用外部 API  配置超时时间（如 30 秒），如果外部请求超时，直接报错。 

### 4.3 正确性设计 (Correctness) 
1. **条件过滤 (WHERE) 与参数的安全下推**：   * 如果用户输入：`MATCH (ne)-[:NEHasKPI]->(target) WHERE target.value > 90`   * **设计要求**：AST 重写时，属于 `target`（虚拟节点）的 `WHERE` 条件**绝对不能**扔给 TuGraph 物理查询执行，否则报语法错误。Visitor 必须将 `value > 90` 解析为过滤算子规则，传递给 Java 层的 HTTP Client，让外部引擎去做条件过滤。

2. **分页与排序语义颠覆**：   * **设计要求**：一旦 Cypher 被切分，物理数据库层的 `ORDER BY` 和 `LIMIT` 将失去全局准确性。   * 引擎必须剥离 AST 中的 `LIMIT/ORDER BY`。物理查询需要拉取**全量**（或较大数量）候选数据集，合并外部数据后，统一在 Java 层的集合 (Collections / Stream API) 中执行最终排序和分页，保证客户端看到的数据逻辑正确。 

3. **数据拼接防重叠**：   * 在使用 `UNION` 操作符时，TuGraph 默认会做全量结果去重（相当于 SQL 的 UNION DISTINCT）。我们的内存聚合器层，必须对最终结果集合计算 Hash 值进行去重，确保行为与原生 Cypher 完全一致。 请分析如上设计文档，分析技术可行性，列出详细的实现计划
